# Canonical Spec Registry

## Status
✅ Defined — optional persistent registry protocol for cross-session consistency

---

## Purpose

Session memory only preserves state inside the current conversation. It cannot prevent the same concept, such as `bookmark`, from producing divergent specs across separate sessions.

For production component libraries, use a persistent **canonical spec registry** when available.
The registry can be any project-level durable source, for example:

- `icon-registry.json`
- `canonical-icon-specs.md`
- a repo `specs/` folder
- the existing Figma component library
- another user-approved persistent icon inventory

This skill can read from or emit entries for a registry if the environment provides one, but it must not claim cross-session consistency is guaranteed when no persistent registry is available.

---

## Registry protocol

### Before generating a new icon

Before Phase 2 for a repeated or common concept, and before Phase 4A for production drawing:

```
1. Check whether a canonical registry or existing platform component library is available.
2. Search by:
   - meta.name, e.g. AijBasicBookmark
   - label, e.g. Bookmark
   - aliases, e.g. favorite, save, mark chapter
   - context, e.g. chapter card, toolbar, nav
3. If an exact canonical spec exists:
   - do not silently regenerate a divergent version
   - propose reuse or controlled variant creation
4. If an adjacent spec exists:
   - match its metaphor family, proportions, radius behavior, density, and naming pattern
5. If no registry is available:
   - continue with normal workflow
   - report that cross-session consistency cannot be guaranteed without a persistent registry
```

### Conflict rule

If the requested icon concept already has a canonical entry but the new direction differs:

```
Ask the user to choose:
A. Reuse the canonical spec
B. Create a named variant with a clear reason
C. Replace/update the canonical spec
```

Do not overwrite a canonical direction silently.

---

## After final approval

After the Figma output passes validation and screenshot review, emit a suggested registry entry for the user or build system to store.

Use this format:

```jsonc
{
  "name": "AijBasicBookmark",
  "label": "Bookmark",
  "aliases": ["favorite", "save chapter", "mark"],
  "context": ["chapter card", "toolbar"],
  "semantic_direction": "bookmark ribbon with rounded bottom notch",
  "visual_elements": ["ribbon outline", "bottom notch"],
  "style": "outline",
  "size": 24,
  "stroke": 2,
  "corner_radius": "4px default, adjusted where noted",
  "uses_local_fill": false,
  "shape_summary": "1 open/closed path, no fills",
  "editable_fidelity": "native editable nodes",
  "preview_fidelity": "matches approved SVG preview",
  "source": "generated by icon-gen-promax skill",
  "status": "approved"
}
```

If the environment supports writing to a registry, update it only after user approval.
If not, include the registry entry in the final report so the user can copy it into their project.

---

## Final report fields

Include these fields in production handoff reports:

```md
- Canonical registry: {checked + reused | checked + new entry emitted | unavailable}
- Cross-session consistency: {covered by registry | not guaranteed without registry}
```

Do not repeat long registry warnings for every icon in a batch. One batch-level note is enough.
